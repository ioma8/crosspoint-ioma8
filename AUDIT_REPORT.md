# CrossPoint Firmware — Full Codebase Audit Report

Generated by static analysis of the full source tree.

---

## 🔴 CRITICAL

### Network & Security

**[NET-C1] Zero authentication on the entire HTTP + WebSocket server**
`src/network/CrossPointWebServer.cpp:94–193`
Every route — file list, download, upload, delete, rename, settings read/write, OTA trigger — is accessible to any device on the LAN with a single `curl` command. No HTTP Basic Auth, token, session, or IP allowlist exists. This makes most other network findings below trivially exploitable.

**[NET-C2] Path traversal in `handleDownload` exposes WiFi credentials**
`src/network/CrossPointWebServer.cpp:488–499`
The handler extracts only the **last path segment** with `lastIndexOf('/')` and checks it against the protected list. A request for `GET /download?path=/.crosspoint/wifi.json` bypasses all guards (the final segment `wifi.json` looks innocent) and streams the credential file to the caller. The WebDAV handler correctly uses `isProtectedPath()` which walks every segment — this handler does not.

**[NET-C3] WebSocket upload writes to any path, including `/.crosspoint/`**
`src/network/CrossPointWebServer.cpp:1295–1315`
The `START:<filename>:<size>:<path>` message handler only checks that the path starts with `/`. It never calls `isProtectedItemName()` or `normalizeWebPath()`. A message `START:wifi.json:256:/.crosspoint` overwrites the WiFi credential store with attacker-supplied data.

**[NET-C4] TLS certificate verification completely disabled for all HTTPS downloads**
`src/network/HttpDownloader.cpp:59, 101`
`setInsecure()` is called unconditionally on every `NetworkClientSecure` instance, disabling all X.509 verification (CA chain, expiry, hostname). Every OPDS feed fetch and book download is silently vulnerable to MITM on any untrusted Wi-Fi network.

**[NET-C5] OTA TLS hostname check disabled — MITM can install arbitrary firmware**
`src/network/OtaUpdater.cpp:107, 196`
`skip_cert_common_name_check = true` is set on both the version-check request and the firmware download. An attacker holding any certificate from a trusted CA can impersonate `api.github.com`, return JSON pointing to a malicious binary, and have it flashed silently.

**[KOR-C1] TLS verification disabled for all KOReader HTTPS sync calls**
`lib/KOReaderSync/KOReaderSyncClient.cpp:44`
`setInsecure()` on every `WiFiClientSecure` instance covers all three API calls (authenticate, getProgress, updateProgress). Combined with KOR-C2, this enables credential harvesting via evil-twin AP.

**[KOR-C2] HTTP Basic Auth plaintext password sent unconditionally, including over plain HTTP**
`lib/KOReaderSync/KOReaderSyncClient.cpp:25–26`
`addAuthHeaders()` calls `http.setAuthorization(username, plaintext_password)` regardless of scheme. Any user whose stored URL lacks `https://` gets their actual password sent in cleartext.

### EPUB / ZIP / Serialization

**[ZIP-C1] Three RISC-V alignment faults in EOCD scanner**
`lib/ZipFile/ZipFile.cpp:247–273`
The EOCD scan decrements the offset by 1 each iteration, meaning 3 of every 4 probes use a misaligned address. Two more unaligned `reinterpret_cast` dereferences read `totalEntries` (16-bit) and `centralDirOffset` (32-bit) at non-aligned offsets. The ESP32-C3 RISC-V core hard-faults on these. Must be replaced with `memcpy` into typed locals (as the `getDataOffset` function in the same file already does correctly).

```cpp
// Current (FAULT: i is unaligned on 3 of every 4 iterations)
if (*reinterpret_cast<uint32_t*>(&buffer[i]) == signature) { ... }
zipDetails.totalEntries     = *reinterpret_cast<uint16_t*>(&buffer[foundOffset + 10]);
zipDetails.centralDirOffset = *reinterpret_cast<uint32_t*>(&buffer[foundOffset + 16]);

// Fix: use memcpy
uint32_t sig; memcpy(&sig, &buffer[i], 4);
uint16_t tot; memcpy(&tot, &buffer[foundOffset + 10], 2);
uint32_t cdo; memcpy(&cdo, &buffer[foundOffset + 16], 4);
```

**[ZIP-C2] Integer overflow → heap OOB write in `readFileToMemory`**
`lib/ZipFile/ZipFile.cpp:311–318`
`inflatedDataSize` is a `uint32_t` from the ZIP central directory (attacker-controlled in a crafted EPUB). When `inflatedDataSize == UINT32_MAX` and `trailingNullByte == true`, `dataSize = inflatedDataSize + 1` wraps to `0`. `malloc(0)` returns non-null, the null-check passes, and `data[UINT32_MAX]` writes one byte ~4 GB off the heap. No upper-bound sanity check exists.

**[SER-C3] Uncapped heap allocation from file-controlled string length in `readString`**
`lib/Serialization/Serialization.h:43–52`
`readString` reads a `uint32_t len` from the cache file and calls `s.resize(len)` with no cap. This function is called from **every** deserialization path (`BookMetadataCache`, `TextBlock`, `Section`, `CssParser`, etc.). A single bit-flip on an SD card that corrupts a 4-byte length field causes the firmware to call `std::terminate` (no-exception build). Additionally, the `file.read()` return value is discarded — on a short read, the string contains uninitialized bytes used as file paths and anchor IDs.

### HAL

**[HAL-C1] RISC-V alignment hazard inside the IRAM panic handler**
`lib/hal/HalSystem.cpp:62`
The panic handler casts `sp` (a raw stack-pointer from the exception frame) directly to `uint32_t*` and dereferences it. In a crash scenario the stack may not be 4-byte aligned. This causes a *second* fault inside the `IRAM_ATTR` handler, entering an infinite panic loop and destroying the crash report.

### Activity Layer

**[ACT-C1] Orphaned FreeRTOS task — handle discarded, no cancel in `onExit()`**
`src/activities/reader/epub/EpubReaderActivity.cpp:814`
`maybeScheduleHomeThumbGeneration()` creates a 12 KB FreeRTOS task but stores the handle in a local variable that is immediately discarded. `onExit()` calls `epub.reset()` with no knowledge of the running task. Rapid book switching stacks up multiple 12 KB orphaned stacks until OOM.

**[ACT-C2] Same orphaned-task pattern in `XtcReaderActivity`**
`src/activities/reader/xtc/XtcReaderActivity.cpp:~170`
Identical issue: handle discarded, `onExit()` calls `xtc.reset()` without cancelling the background task.

**[ACT-C3] `char name[500]` local stack variable in `FileBrowserActivity::loadFiles()`**
`src/activities/home/FileBrowserActivity.cpp`
500 bytes on the stack violates the project's hard 256-byte per-function limit for the ESP32-C3.

**[ACT-C4] `char name[500]` local stack variable in `SleepActivity::renderCustomSleepScreen()`**
`src/activities/boot_sleep/SleepActivity.cpp:44–60`
Same violation. Called from `onEnter()` which is already deep in the activity manager call chain.

## 🟠 HIGH

### Network

**[NET-H1] `handleFileListData` — no path normalization, `/.crosspoint` listing allowed**
`src/network/CrossPointWebServer.cpp:425–440`
`?path=/.crosspoint` is only stripped of a trailing slash; it is never normalized or checked against protected names. `scanFiles("/.crosspoint", ...)` opens that directory and iterates its children. Files inside (e.g. `wifi.json`) have names that do not start with `.`, so the `shouldHide` guard does not hide them. The caller receives a complete JSON listing of every file in the credential store directory.

**[NET-H2] HTTP multipart upload filename not sanitized — `../` traversal**
`src/network/CrossPointWebServer.cpp:595`
`upload.filename` is taken verbatim from the multipart `Content-Disposition` header. A filename of `../secret.bin` combined with a legitimate upload path produces a file outside the intended directory.

**[NET-H3] WebSocket upload filename not sanitized — `../` traversal**
`src/network/CrossPointWebServer.cpp:1280`
`wsUploadFileName` is extracted from the text payload without checking for `..` or `/`. `START:../../root.bin:1024:/some/path` writes a file two levels up from `wsUploadPath`.

**[NET-H4] `handleDelete` only checks the last path segment**
`src/network/CrossPointWebServer.cpp:1003–1020`
The same logic flaw as NET-C2. A request body `["/.crosspoint/wifi.json"]` deletes the credential file without obstruction.

**[NET-H5] 4 KB `uint8_t buffer[4096]` on the stack in `handleDownload`**
`src/network/CrossPointWebServer.cpp:532–533`
Hard limit is 256 bytes per function per CLAUDE.md. Stack overflow risk during large file downloads.

**[NET-H6] 4 KB `uint8_t buf[4096]` on the stack in `WebDAVHandler::handleCopy`**
`src/network/WebDAVHandler.cpp:632`
Same violation, fired on every WebDAV sync operation.

**[NET-H7] `wsServer` not null-checked after `new WebSocketsServer(...)`**
`src/network/CrossPointWebServer.cpp:177–180`
On OOM, `wsServer->begin()` is called on a null pointer.

**[NET-H8] `writeToStream` return value ignored in `fetchUrl(Stream&)` — silent data loss**
`src/network/HttpDownloader.cpp:86–89`
The `Stream` overload discards the return value and unconditionally returns `true`. Callers processing partial OPDS XML silently see success.

**[KOR-H1] `http.getString()` with no max size — heap exhaustion from rogue server response**
`lib/KOReaderSync/KOReaderSyncClient.cpp:91`
The entire response body is buffered into an Arduino `String` with no size limit. A rogue or compromised sync server can exhaust all available heap.

### EPUB / ZIP

**[ZIP-H1] Memory leak: output buffer not freed on secondary `malloc` failure**
`lib/ZipFile/ZipFile.cpp:319–332`
`data` is allocated; if the subsequent `deflatedData = malloc(...)` fails, `data` is never freed. Permanent heap loss per failed decompression.

**[ZIP-H2] Uninitialized stack variables used after ignored `file.read()` failures**
`lib/ZipFile/ZipFile.cpp:63–76` (and `~120–140`)
`uint16_t nameLen, m, k` are declared but never zero-initialized. If `file.read()` returns 0 (SD error, truncated ZIP), all three retain garbage values used in `seekCur` and `file.read(itemName, nameLen)`.

**[EPUB-H3] No `currentPage` bounds check — negative value wraps `size_t` offset**
`lib/Epub/Epub/Section.cpp:166–178`
`currentPage` is a public `int`. A value of `-1` makes `sizeof(uint32_t) * currentPage` wrap to `0xFFFFFFFC` as a `size_t`, seeking near the file start and reading the header as page data. An out-of-range positive value reads beyond the LUT.

**[EPUB-H4] File-controlled element `count` drives unbounded deserialization loop**
`lib/Epub/Epub/Page.cpp:84–103`
A corrupt cache with `count = 65535` drives 65535 iterations of `PageLine::deserialize`, each calling `TextBlock::deserialize`, each calling `readString` with uncapped lengths. Full RAM exhaustion.

**[EPUB-H5] Word-count cap bypassed by per-word uncapped string length**
`lib/Epub/Epub/blocks/TextBlock.cpp:60–82`
`wc` is capped at 10,000 but each word's `readString` length field is uncapped. `wc=100` x `len=4,000,000` = 400 MB of allocation attempts → `std::terminate`.

**[EPUB-H6] `FsFile` handle leaked on two error paths in `parseTocNavFile`**
`lib/Epub/Epub.cpp:228–242`
`tempNavFile` is opened but not closed on the `!navParser.setup()` and `!navBuffer` error returns. Exhausts SdFat file descriptor slots on repeated failures.

### GfxRenderer

**[GFX-H1] Unchecked `new[]` in all three ditherer constructors**
`lib/GfxRenderer/BitmapHelpers.h:30–36, 102–108, 199–203`
With `-fno-exceptions`, `operator new` returns `nullptr` on OOM. If the first allocation succeeds but the second returns null, the object is half-constructed and the first `processPixel` call writes to a null pointer — hard fault.

**[GFX-H2] Outer ditherer `new` not null-checked; silent degradation with no logging**
`lib/GfxRenderer/Bitmap.cpp:167–173`
If `new AtkinsonDitherer(...)` returns null, the ditherer is silently skipped and cover images render without dithering with no error indication.

**[GFX-H3] `storeBwBuffer()` false return not marked `[[nodiscard]]`; unchecked callers leave framebuffer corrupt**
`lib/GfxRenderer/GfxRenderer.cpp:1259–1284`
If `storeBwBuffer()` fails mid-way, it correctly calls `freeBwBufferChunks()` and returns `false`. If the caller ignores this and proceeds to overwrite the framebuffer with grayscale data, `restoreBwBuffer()` bails silently without calling `display.cleanupGrayscaleBuffers()`, leaving the display permanently corrupted.

**[GFX-H4] Arduino `String` heap allocations in main `loop()`**
`src/main.cpp:332–334`
Two `String` objects heap-allocate on every loop iteration where serial bytes arrive. Continuous heap fragmentation in the main task.

### Activity Layer

**[ACT-H1] OTA `FINISHED` state has no handler — device stuck until physically power-cycled**
`src/activities/settings/OtaUpdateActivity.cpp`
After a successful OTA download, `state = FINISHED` is set but no branch in `loop()` matches it. `SHUTTING_DOWN` (which calls `ESP.restart()`) is never assigned, so the newly installed firmware never activates without a manual power cycle.

**[ACT-H2] `std::shared_ptr<Epub>` violates single-core policy**
`src/activities/reader/epub/EpubReaderActivity.h`
CLAUDE.md explicitly prohibits `std::shared_ptr` due to atomic refcount overhead on single-core RISC-V. This propagates to 4–5 sub-activities and a background task with fully bounded lifetimes.

**[ACT-H3] `std::shared_ptr<Xtc>` violates single-core policy**
`src/activities/reader/xtc/XtcReaderActivity.h`
Same issue as ACT-H2.

**[ACT-H4] SPIFFS write on every button press in `SettingsActivity::toggleCurrentSetting()`**
`src/activities/settings/SettingsActivity.cpp`
`SETTINGS.saveToFile()` is called unconditionally on every `Confirm` press with no value-change guard. The `loop()` Back handler adds a second unconditional save, producing **two** SPIFFS writes per toggle.

**[ACT-H5] Same unguarded SPIFFS write in `StatusBarSettingsActivity::handleSelection()`**
`src/activities/settings/StatusBarSettingsActivity.cpp:85–110`
Six individual boolean/enum toggles each write to SPIFFS with no guard.

**[ACT-H6] Blocking HTTP on main task in `KOReaderSyncActivity` — no WDT reset**
`src/activities/reader/shared/KOReaderSyncActivity.cpp:100–135`
Both `performSync()` and `performUpload()` make blocking HTTP calls from the Arduino main task without `esp_task_wdt_reset()`. A slow or unreachable server triggers a hardware watchdog reset mid-sync.

## 🟡 MEDIUM

### Network

**[NET-M1] Hardcoded XOR key `"CrossPoint"` in every release binary**
`src/app/WifiCredentialStore.cpp:24`
The legacy migration key (`0x43 0x72 0x6F 0x73 0x73 0x50 0x6F 0x69 0x6E 0x74`) is compiled into every release. Any `wifi.bin` extracted from an SD card (e.g., via NET-C2) can be decoded with a trivial single-pass XOR loop.

**[NET-M2] `addCredential` writes to SD even when password is unchanged**
`src/app/WifiCredentialStore.cpp:112–115`
Missing `if (cred->password == password) return true;` guard. Every reconnection attempt triggers an unnecessary SD sector write.

**[NET-M3] Raw `SETTINGS` struct offset read — `strlen` walks past struct boundary**
`src/network/CrossPointWebServer.cpp:1128–1132`
ArduinoJSON calls `strlen` on a raw pointer computed as `&SETTINGS + s.stringOffset`. A misconfigured entry leaks adjacent DRAM (heap metadata, other settings) into the web response.

**[NET-M4] Raw `SETTINGS` struct offset write via unauthenticated POST — OOB write risk**
`src/network/CrossPointWebServer.cpp:1209–1214`
`s.stringOffset` is not bounds-checked against `sizeof(CrossPointSettings)`. With NET-C1 in place (no auth), anyone on the network can trigger an OOB write into adjacent RAM.

**[NET-M5] `std::string` allocated per STRING setting on every `handlePostSettings` call**
`src/network/CrossPointWebServer.cpp:1209`
Violates the project string policy. Every POST iterates all settings and heap-allocates a `std::string` for each string-type field.

**[NET-M6] WebSocket upload `size` field: `toInt()` saturates at `LONG_MAX` — trivial unauthenticated DoS**
`src/network/CrossPointWebServer.cpp:1290`
A size string longer than 10 digits saturates at `LONG_MAX`, making the overflow guard never trip and causing the device to write data until the SD card is full.

**[NET-M7] `handleCreateFolder` parent path not normalized — dirs creatable inside `/.crosspoint/`**
`src/network/CrossPointWebServer.cpp:752–764`
`path=/.crosspoint&name=injected` creates `/.crosspoint/injected` without restriction.

**[NET-M8] `isNewer()` returns `true` when only the `v` prefix differs — spurious OTA install loop**
`src/network/OtaVersion.h:75–78`
If the GitHub tag is `v1.2.3` and the compile-time constant is `1.2.3`, `parseSemverPrefix` produces equal structs but the trailing string comparison sees a difference and triggers an update.

**[KOR-M1] Plaintext `"password"` JSON field silently accepted as fallback**
`lib/KOReaderSync/KOReaderCredentialCodec.cpp:28–33`
A crafted or accidental `koreader.json` with a `"password"` key instead of `"password_obf"` is silently accepted, bypassing obfuscation entirely.

### EPUB / ZIP / PDF / PNG

**[ZIP-M1] All central-directory `file.read()` return values silently ignored**
`lib/ZipFile/ZipFile.cpp:60–90, 115–160, 290–330`
A truncated or damaged ZIP causes stale/zero values to propagate as compression method, sizes, filename length, and local header offset with no error indication to the caller.

**[ZIP-M2] `out.write()` failure ignored in STORED streaming path**
`lib/ZipFile/ZipFile.cpp:374–383`
Returns `true` even though the output was silently truncated by a full SD card or stream error.

**[EPUB-M3] Cache image dimensions uncapped — corrupt cache can cause watchdog-timeout loop**
`lib/Epub/Epub/blocks/ImageBlock.cpp:38–45`
`cachedHeight = 65535` drives 65535 SD sector reads inside the render loop, triggering a watchdog reset. The dimension-mismatch guard only applies when values are more than ±1 pixel off expected.

**[EPUB-M4] `int16_t` cast overflow for EPUBs with >32 767 spine items**
`lib/Epub/Epub/BookMetadataCache.cpp:248`
`spineIndex = static_cast<int16_t>(i)` wraps negative for `i >= 32768`, silently corrupting the TOC for the second half of a large book's spine.

**[EPUB-M5] `lut` vector missing `reserve()` during chapter indexing**
`lib/Epub/Epub/Section.cpp:118–125`
~7 reallocation events per 200-page chapter, each allocating, copying, and freeing on an already-fragmented heap.

**[EPUB-M6] Anchor map `count` + `readString` uncapped from corrupt cache**
`lib/Epub/Epub/Section.cpp:200–215`
Compounds SER-C3 and EPUB-H4: 65535 iterations of `readString` each with uncapped allocation.

**[PDF-M1] `firstObj + count` unsigned overflow bypasses xref guard**
`lib/Pdf/Pdf/XrefTable.cpp:374–376, 416–419`
On 32-bit ESP32-C3, if `firstObj ≈ ULONG_MAX - 99999` and `count <= 100000`, their sum wraps below 2,000,000 and passes the range check, forcing a multi-megabyte file seek.

**[PNG-M2] `chunkLen + 4` uint32 overflow when skipping large PNG chunks**
`lib/PngToBmpConverter/PngToBmpConverter.cpp:219–221`
A crafted PNG with `chunkLen = 0xFFFFFFFF` causes `chunkLen + 4` to wrap to 3, misaligning all subsequent chunk parsing.

### GfxRenderer / HAL / Core

**[HAL-M1] `__wrap_abort` missing `IRAM_ATTR`**
`lib/hal/HalSystem.cpp:104`
`__wrap_abort` intercepts `abort()`, `std::terminate`, and assert failures. Without `IRAM_ATTR`, a call during SPI flash operations (SPIFFS write, OTA) faults because the instruction cache is suspended. `__wrap_panic_abort` in the same file is correctly marked — `__wrap_abort` is not.

**[GFX-M1] Floyd-Steinberg "serpentine" scanning is logically broken**
`lib/GfxRenderer/BitmapHelpers.h:228–235`
The calling loop always iterates left-to-right, but the "reverse" branch distributes the 7/16 error component to an already-processed pixel slot, silently discarding it every other row. Results in visible banding artifacts on grayscale cover images.

**[GFX-M2] `uint16_t byteIndex` narrowing from `int` arithmetic in `drawPixel`**
`lib/GfxRenderer/GfxRenderer.cpp:218`
Safe for the current 800×480 display (max value 47,999), but silently wraps if `DISPLAY_HEIGHT` ever reaches 656+. Should be `uint32_t` to match `BUFFER_SIZE`.

**[MAP-M1] No bounds check on `kSideLayouts[sideLayout]` array access**
`src/app/MappedInputManager.cpp:20`
`sideLayout` is cast directly from `SETTINGS.sideButtonLayout`. The array has exactly 2 entries; a value >= 2 is undefined behaviour that reads arbitrary flash/DRAM and feeds it as a physical button index to `InputManager`.

**[MAP-M2] Remapped front-button hardware index passed to `InputManager` without validation**
`src/app/MappedInputManager.cpp:28–38`
`SETTINGS.frontButtonBack/Confirm/Left/Right` are passed directly. Valid range is 0–3; a corrupt or directly-written value >= 4 accesses an OOB button slot at runtime.

**[SET-M1] `CrossPointSettings::saveToFile()` has no value-change guard**
`src/app/CrossPointSettings.cpp:71–74`
CLAUDE.md requires a guard before every settings write. Any caller that invokes `SETTINGS.saveToFile()` per button press causes gratuitous SD writes.

**[SET-M2] `RecentBooksStore::addBook` / `updateBook` — unconditional SD writes per book open**
`src/app/RecentBooksStore.cpp:41, 57`
Two SD writes (add + metadata update) per reading session with no value-change guard.

**[SET-M3] `CrossPointSettings` version check uses `!=` instead of `>`**
`src/app/CrossPointSettings.cpp:~114`
`CrossPointState` correctly uses `>` with per-field migration guards. Settings uses strict equality — a firmware downgrade silently resets all user settings to defaults with no warning or migration.

**[OPDS-M1] `OpdsParser::flush()` dereferences null `parser` pointer after any write-time error**
`lib/OpdsParser/OpdsParser.cpp:271` + `lib/OpdsParser/OpdsStream.cpp:12`
`write()` frees `parser` and sets it to `nullptr` on XML allocation failure. The `OpdsParserStream` destructor always calls `flush()`, which then calls `XML_Parse(nullptr, ...)`. Every malformed OPDS document triggers a guaranteed crash.

**[OPDS-M2] Unbounded `currentText.append()` in XML character-data callback**
`lib/OpdsParser/OpdsParser.cpp:184`
A malicious OPDS server can send a title/author/id element with megabytes of content in a single text node, exhausting all 380 KB of heap. The `entries` vector is also uncapped.

**[KOR-M2] `std::shared_ptr<Epub>` in `ProgressMapper` — atomic overhead on single-core RISC-V**
`lib/KOReaderSync/ProgressMapper.cpp:7`
Violates CLAUDE.md. Should be `const Epub&` or `const Epub*`.

**[IMG-M1] `new uint32_t[outWidth]` without null-check in JPEG scaler**
`lib/JpegToBmpConverter/JpegToBmpConverter.cpp:355–356`
With `-fno-exceptions`, `operator new[]` returns `nullptr` on OOM. Subsequent `rowAccum[outX] += sum` is a null-pointer store fault on a 380 KB device processing large cover images.

**[IMG-M2] `rowCount` typed as `uint16_t*` in PNG scaler — integer overflow under moderate downscaling**
`lib/PngToBmpConverter/PngToBmpConverter.cpp:637–641`
JPEG correctly uses `uint32_t*`. For a 2048×3072 PNG downscaled to 1×1, accumulation reaches 6,291,456 — 96x the uint16 limit — wrapping to zero and producing a solid-black output. Even moderate 4x downscaling can trigger overflow.

## 🔵 LOW

**[NET-L1] `strtol` overflow not checked in `parseSemverPrefix`**
`src/network/OtaVersion.h:30–44`
`std::strtol` sets `errno = ERANGE` on overflow but none of the three calls check it. A release tag of `"99999999999.0.0"` silently coerces `major` to `INT_MAX`.

**[NET-L2] Partial upload file not cleaned up on write-error path**
`src/network/CrossPointWebServer.cpp:663–668`
The `UPLOAD_FILE_ABORTED` path correctly removes the partial file; the write-error path does not. Orphaned partial files persist on the SD card.

**[NET-L3] `saveToFile()` return value silently discarded in `WifiCredentialStore`**
`src/app/WifiCredentialStore.cpp:~161, ~170`
`setLastConnectedSsid()` and `clearAll()` both discard the `saveToFile()` return value. A failed write leaves in-memory state out of sync with on-disk state across reboots.

**[NET-L4] `std::function` callback in `scanFiles` heap-allocates a closure on every file listing**
`src/network/CrossPointWebServer.h:83`
Per CLAUDE.md, `std::function` adds ~2–4 KB per unique signature and heap-allocates its closure. A function pointer + context struct would achieve the same with zero heap cost.

**[NET-L5] Five global Arduino `String` objects for WebSocket state persist for firmware lifetime**
`src/network/CrossPointWebServer.cpp:33–42`
Their internal heap buffers are allocated on first write and never freed, contributing to long-term heap fragmentation.

**[NET-L6] `new WebDAVHandler()` result not null-checked before `addHandler`**
`src/network/CrossPointWebServer.cpp:168`
A null pointer registered as a handler can be dereferenced during `handleClient()` calls.

**[NET-L7] `.davtmp` suffix can exceed FAT 255-char path limit, leaving orphan temp files**
`src/network/WebDAVHandler.cpp:80, 97, 112`
A source path of 249+ characters creates an invalid temp path. The cleanup `Storage.remove(tempPath.c_str())` then fails silently, leaving the partial `.davtmp` file permanently.

**[GFX-L1] `prevRowY` not reset in `Bitmap::rewindToData()`**
`lib/GfxRenderer/Bitmap.cpp:276–283`
After a rewind, `prevRowY` retains its end-of-image value. Noise-dithering mode (`USE_NOISE_DITHERING`) would use the wrong y-coordinate in `quantize()`. Currently harmless as the mode defaults to `false`.

**[GFX-L2] `createBmpHeader` does not validate sign or bounds of `width`/`height`**
`lib/GfxRenderer/BitmapHelpers.cpp:110–114`
A negative or zero `width`/`height` produces a large positive `rowSize`/`imageSize` after unsigned wrap, writing a malformed BMP header.

**[GFX-L3] `wrappedText()` allocates a temporary `std::string` per word in layout loop**
`lib/GfxRenderer/GfxRenderer.cpp:991–1053`
~200 `malloc`/`free` cycles of ~20 bytes each per paragraph. Creates long-lived fragmentation holes between larger allocations.

**[GFX-L4] O(n²) linear scan in `Utf8CodepointCollector::add`**
`lib/GfxRenderer/Utf8CodepointCollector.cpp:28–36`
O(n) per `add()` call; called for every character during the font pre-warm scan. A sorted array + binary search or a bitset would give O(log n) or O(1) at no RAM cost.

**[HAL-L1] `HalPowerManager::Lock` constructor calls `xSemaphoreTake` on potentially-null mutex**
`lib/hal/HalPowerManager.cpp:88`
`modeMutex` is initialized in `begin()`. If `Lock` is ever constructed before `begin()` (global or early static constructor), `xSemaphoreTake(nullptr, portMAX_DELAY)` triggers a FreeRTOS assertion or silent hang.

**[MAIN-L1] Raw `HalGPIO::BTN_DOWN` bypasses `MappedInputManager` in screenshot hotkey**
`src/main.cpp:352–356`

**[MAIN-L2] `unsigned long → uint16_t` truncation in boot-time power-button calibration**

`millis()` is 32-bit on ESP32. If `setup()` takes > 65.535 s (e.g., SD FSCK), the upper 16 bits are discarded, producing an incorrect calibration value.

`deflatedDataSize` is `uint32_t`; `%d` expects a signed `int`. Values above `INT_MAX` print as negative. Use `%u` or `PRIu32`.
